2.3 Übung

Lesen Sie das Kapitel 3.9. Annotationen in dem Buch "Java ist auch eine Insel" von Christian Ullenboom und lösen Sie folgendes Problem:

Gegeben sei folgende Implementierung der Klasse Zeit, die noch den @Override-Fehler enthält. Darin soll die equals-Methode die gleichnamige Methode aus java.lang.Object überschreiben. Korrigieren Sie die Methode, so dass das Überschreiben gelingt:


class Zeit {
    float time;

    @Override
    boolean equals(Zeit other) {
        return this.time == other.time;
    }
}

Bearbeiten Sie die Projektaufgabe aus der ersten Woche (Übung 0), diesmal konsequent testgetrieben:

In einem Projekt gibt es verschiedene Aktivitäten, die teilweise aufeinander aufbauen und voneinander abhängig sind. Jede Aktivität hat einen Namen. In einem Array werden die Abhängigkeiten zwischen den Aktivitäten gespeichert (binäre "vor"-Relation, z.B. A "vor" C). Das Array 

new String[][]{{"A","C"},{"C","D"},{"B","C"}}
bedeutet z.B., dass die Aktivität A vor der Aktivität C stattfinden muss, C vor D, B vor C. Zwischen A und B gibt es allerdings keine Reihenfolgevorschrift und beide Aktivitäten können in einer beliebigen Reihenfolge ablaufen. Mit der Methode 
isWellSorted(String[] sequence)
soll geprüft werden, ob 
sequence
eine erlaubte Reihenfolge ist. Einen ersten unfertigen Versuch einer Implementierung finden Sie hier.

Project.java


package se_00_ueb;
import java.util.Arrays;

public class Project {
    String[][] order;
    public Project(String[][] order) {
        this.order = order;
    }

    public boolean isWellSorted(String[] sequence) {
        for (int i = 0; i < sequence.length - 1; i++) {
            if (sequence[i].compareTo(sequence[i + 1]) > 0) return false;
        }
        return true;
    }
}

ProjectTester.java


package se_00_ueb;
import static org.junit.Assert.*;
import org.junit.Test;

public class ProjectTester {
    Project example = new Project(
            new String[][] {{"A", "C"}, {"C", "D"}, {"B", "C"}}
    );

    @Test
    public void test() {
        assertTrue(example.isWellSorted(new String[] {"A", "B", "C", "D"}));
        assertFalse(example.isWellSorted(new String[] {"D", "C", "B", "A"}));
    }
}

Denken Sie sich weitere mögliche Sequenzen aus und testen Sie diese. Schreiben Sie selbst den Test zuerst, lassen Sie den Test laufen ( ⇒ Roter Balken ) und lassen Sie danach Ihre Partnerin / Ihren Partner die isWellSorted - Methode erweitern ( ⇒ Grüner Balken ). Prüfen Sie mit Ihrem Test, ob die Methode korrekt implementiert wurde. Finden Sie Testfälle, die von der Methode noch nicht korrekt abgedeckt wurden. Entwickeln Sie testgetrieben Test und Methode weiter wie oben beschrieben, immer Test zuerst und danach Weiterentwicklung der Methode. Benennen Sie Ihre Methoden nach der Namenskonvention A_B_C mit (1.) der zu testenden Methode A, (2.) dem Use Case / Test Szenario B, (3.) und dem erwarteten Verhalten wie in der Vorlesung besprochen.

Aufgabe 4: Beliebige Projekte (6 Punkte)

In Aufgabe 3 wurde das Projekt definiert durch die Reihenfolgevorschriften A ⇒ C, C ⇒ D, B ⇒ C. Denken Sie sich andere binäre Reihenfolgevorschriften aus und setzen Sie Ihre testgetriebene Entwicklung aus Aufgabe 3 fort, bis Sie einen universellen Algorithmus für beliebige Projekte mit binären Reihenfolgevorschriften und beliebige Sequenzen haben.

Entwickeln Sie konsequent testgetrieben: immer Test zuerst und danach Weiterentwicklung der Methode. Benennen Sie Ihre Methoden nach der Namenskonvention A_B_C mit (1.) der zu testenden Methode A, (2.) dem Use Case / Test Szenario B, (3.) und dem erwarteten Verhalten C wie in der Vorlesung besprochen.

Verstricken Sie sich dabei nicht in selbst-verschuldeter Komplexität. Vereinfachen Sie immer wieder Ihre Implementierung in den Refaktorisierungsschritten. Schreiben Sie auf, was in Ihrem Entwicklungsprozess wichtig war für

  1. Analyse (Verstehen der Problemstellung)
  2. Design (Architektur der Lösung, Grundlegende Entscheidungen)
  3. Implementierung
  4. Validierung: Nicht nur Test der Implementierung, sondern auch Reflexion, ob die Problemstellung wirklich richtig verstanden wurde und ob die Design-Entscheidungen auch die Vorteile eingebracht haben, die man sich davon versprochen hat.
